Current State

What exists in the system today

Prompt system + local DB + analytics engine

Longevity OS already behaves like a layered local product rather than a single assistant prompt. The stack is real: a prompt orchestrator routes work to domain prompts, those prompts read and write a structured SQLite schema, and the modeling layer computes trends, anomalies, correlations, and trial analytics over that stored state.

Ownership

What each subsystem owns today

Subsystem Owns Depends on Main handoff
SKILL.md Intent routing and response framing. Department prompts, runtime shell access, database path. Hands structured tasks into prompt modules.

Ranked Findings

Where the actual gaps are

Ranked by demo risk
1

Runtime path drift could split writes and reads across different SQLite files

Highest risk

The codebase had separate absolute path literals across scripts, modeling modules, and prompt docs. That makes a demo look fine while different entrypoints are reading and writing different databases.

2

No named top-level demo reset workflow existed

Operator risk

The repo had a deterministic data generator, but not a single command that reset, seeded, and verified the expected hero scenes before a rehearsal or recording.

3

Read paths grouped timestamped rows by SQLite UTC day instead of the local calendar day

Data integrity risk

The seeded demo showed this clearly: diet rows spilled into the wrong day because `DATE(timestamp)` reinterpreted offset timestamps in UTC.

4

The README and prompt-side setup story drifted from the real Python operator workflow

Review and onboarding risk

The documentation described paths and initialization behavior that were no longer the best source of truth. That creates operator confusion even if the code itself works.

5

Prompt-driven write paths still lack regression coverage

Still open

This branch did not prove that casual meal logging or multi-metric logging always create the correct rows. The demo infrastructure is stronger, but prompt-behavior verification is still missing.

6

Some capability surfaces are still incomplete or environment-sensitive

Known limitation

`scripts/import_apple_health.py` is still a stub, and the full modeling CLIs require scientific Python packages in the shell environment.

Branch Delta

What this branch actually changed

Grouped by workstream, not by file list

Workstream 1: runtime path unification

This is why so many files moved. One small logical fix touched many entrypoints because each one had its own idea of where the database lived.

  • Added paths.py as the runtime path source of truth.
  • Rewired data/db.py, setup, backup, migrate, export, lab import, Apple Health stub, demo seed, and modeling modules to resolve paths through it.
  • Gap addressed: ranked gap 1.

Workstream 2: explicit demo reset workflow

The repo now has a real top-level operator command instead of relying on people to know the seeding script’s side effects.

  • Added scripts/demo_reset.py.
  • Clarified scripts/generate_demo_data.py with --skip-reset.
  • Verification now checks row counts plus the hero protein-sleep insight and trial records.
  • Gap addressed: ranked gap 2.

Workstream 3: read-path local-day fix

The branch changed the analytics read paths to group by the local date prefix stored in the timestamp string rather than letting SQLite reinterpret it in UTC.

  • Touched modeling/engine.py, modeling/patterns.py, and modeling/causal.py.
  • Observed effect: seeded diet coverage now ends on 2026-03-31 instead of spilling into 2026-04-01.
  • Gap addressed: ranked gap 3.

Workstream 4: review and operator docs

The docs pass was meant to make the repo understandable as a product review surface, not just a codebase.

  • Updated README.md to explain the demo reset workflow and what the seed command actually does.
  • Updated SKILL.md so prompt-side setup guidance reflects the Python operator flow.
  • Replaced the markdown architecture note with this HTML deep dive.
  • Gap addressed: ranked gap 4.
Short answer to the “why 16 files?” question Most of the file count comes from one cross-cutting fix: every script and analytics entrypoint that could touch SQLite needed to stop hard-coding its own runtime path.

Verification

How to test this branch

Fast manual confidence

1. Static Python sanity

Make sure the rewired entrypoints still parse.

python3 -m compileall paths.py data scripts modeling

2. End-to-end operator smoke

Run the demo reset flow in a temporary data directory so it cannot pollute your normal local state.

LONGEVITY_OS_PROJECT_DIR=/tmp/longevity-os-demo-test-2 python3 scripts/demo_reset.py
  • Expected: setup succeeds.
  • Expected: seed succeeds.
  • Expected: diet date coverage ends on 2026-03-31.
  • Expected: protein-sleep insight and the two trial records are verified.

3. README-level operator check

Read the README as if you were a fresh operator and confirm it points you at demo_reset.py rather than requiring hidden repo knowledge.

4. What was not fully tested in this shell

The full modeling CLI flows were not executed here because the shell environment does not currently have numpy installed.

  • The local-day fix in analytics was validated through the seeded DB coverage change.
  • Future validation should run weekly report and trial analysis commands in an environment with scientific Python dependencies installed.
