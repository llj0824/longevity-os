Telegram Demo Diagnosis

The bot sounds right, but it does not yet behave like the product.

The gap is not missing demo data. The gap is that Telegram messages are still hitting a longevity-flavored reply layer more reliably than they are hitting the real logging-reporting-trial workflow behind the product story.

349 diet entries already in the database
571 body-metric rows already available
2 real trials already stored, including an active creatine trial
Current State

What is happening now

Telegram message arrives

The transport path is mostly working. The bot receives messages and replies.

Skill tone activates

The bot often answers in a longevity voice after /longevity.

Structured app behavior is inconsistent

Meal logging, report generation, and trial flows do not consistently hit the real database-backed workflow.

User sees plausible prose

The reply feels intelligent, but it often looks more like coaching text than a real application output.

Target State

What the product needs to do

Telegram message arrives

Input is parsed into a concrete domain intent like meal log, metric log, report, insight, or trial request.

Workflow executes

The runtime performs structured reads and writes against SQLite and dispatches multi-agent logic where needed.

DB-backed result is produced

The response includes real logged rows, real trend numbers, real insights, and real trial state.

Telegram output demos cleanly

The message is concise, visual, and trustworthy enough to record in a single pass.

What I think is going on

1. Data is not the problem

The database already contains meaningful diet, metrics, insight, and trial data. The wow-demo facts are already there.

2. Prompt richness is not the problem

The longevity skill prompt clearly describes a structured medical-academy style system with database writes, queries, and trial orchestration.

3. Integration discipline is the problem

The runtime is still too willing to answer with plausible prose when it should be forced through concrete intent dispatch, DB reads, DB writes, and explicit trial pipeline steps.

Current vs Target Gap

Current

“I look like a longevity assistant.”

Target

“I am a real application layer over a longevity database and trial engine.”

Demo risk

If we record before closing this gap, the video will overclaim product behavior and underdeliver on the most important scenes.

Ranked Fix Order

  1. Make skill loading deterministic. Until the longevity skill is discovered reliably, every other fix sits on shaky ground.
  2. Force Telegram into the real longevity workflow. The bot needs to do more than change tone after /longevity.
  3. Get meal logging and metric logging writing to SQLite. This turns the first half of the demo from theater into evidence.
  4. Make reports and insight answers read real stored data. This is where the product becomes obviously different from a generic chatbot.
  5. Wire the trial design and trial status scenes end to end. This is the differentiator and needs to be genuinely live.
  6. Tighten Telegram formatting and rehearsal flow. Polish only matters after the outputs are truly correct.

Short version: the bot already has the voice and the data. What it still lacks is strict workflow execution.
